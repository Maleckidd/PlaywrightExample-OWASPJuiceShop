# Playwright E2E Tests — OWASP Juice Shop (AI-migrated from Cypress)

This repository is the **raw output of an AI-driven test migration experiment**: an existing
Cypress test suite for [OWASP Juice Shop](https://github.com/juice-shop/juice-shop) migrated to
**Playwright + TypeScript** entirely by an AI coding agent.

> **Read this first.** The code here is intentionally left almost exactly as the AI produced it —
> mistakes included. This repo is evidence for an article about what AI-assisted test migration
> *actually* delivers, not a showcase of hand-polished code. The honest results are in
> [What actually works](#what-actually-works) and [What broke](#what-broke).

📝 **Article:** _link will be added when published._

## How this was made

- **Migrated by:** [opencode](https://opencode.ai) wired up to GitHub Copilot, running Claude Opus
  4.6 (July 2026). 100% of the code in
  this repo — page objects, specs, config, and the AI's own migration docs — was AI-generated.
  Exactly **two** manual fixes were applied afterwards (the popup-dismissal helper in
  `src/pages/FirstVisitPopups.ts` and one wrong link id in `src/pages/RegistrationPage.ts`) to
  measure how much targeted repairs unlock — see
  [Test results](#test-results-the-honest-part). Everything else is untouched.
- **Source:** a Cypress 10 (JavaScript) suite with page objects and 1/2/3-star OWASP Juice Shop
  challenge tests.
- **Frozen state:** commit `4347f4b` is the untouched AI output. To let the article link a stable
  reference, tag it (see [Reproducing the frozen state](#reproducing-the-frozen-state)) so later
  fixes never rewrite the experiment.
- **AI-generated docs kept as-is:** [MIGRATION_GUIDE.md](./MIGRATION_GUIDE.md),
  [MIGRATION_SUMMARY.md](./MIGRATION_SUMMARY.md), [QUICK_REFERENCE.md](./QUICK_REFERENCE.md) and
  [AGENTS.md](./AGENTS.md) were written by the AI during the migration. They are preserved verbatim,
  **including their factual drift** — e.g. MIGRATION_SUMMARY claims "4 suites / 34 scenarios" while
  the repo actually contains 7 spec files and 89 scenarios per browser. That gap is itself part of
  the story.

### The prompt

The prompt given to the agent, translated to English (the original was written in Polish — the
verbatim original is collapsed below). The prompt itself was **generated by a model, tailored
specifically for the model performing the migration** — AI wrote the contract for AI. Note the
requirements the output later ignored anyway, like the preference for `getByRole`/`getByLabel`
over CSS selectors:

```text
Role: Act as a Senior QA Automation Engineer specializing in Playwright/TypeScript
and Cypress/JavaScript.
Context: I have a framework written in Cypress (JS) and I am working on porting it to
Playwright (TS).
Task: Lay out a strategy and plan for migrating the framework from Cypress (JS) to Playwright (TS)
Technical requirements:
The Playwright framework must work correctly.
The code must follow standards.
Use modern Playwright locators (prefer getByRole, getByText, getByLabel over CSS/XPath selectors).
Stay consistent with the existing style (e.g. camelCase, use of semicolons).
Ensure test readability and resilience (flakiness-free).

Step 1: Initialization and TS Configuration
In Playwright, TypeScript is a first-class citizen. Forget cypress.json — your heart is
playwright.config.ts.
Setup: npm init playwright@latest (choose TypeScript).
Standards: Configure tsconfig.json and eslint for Playwright rules (e.g. enforcing await).
Folder structure:
tests/ - .spec.ts files
pages/ - Page Object Models
utils/ - helper functions
fixtures/ - test data and custom fixtures.

Step 2: Mapping the Page Object Model (POM) logic
In Cypress, cy.commands are often overused. In Playwright we go for a classic but modern POM
with strong typing.

Step 3: Implementing modern locators
As required, we move away from cy.get('.btn-submit') towards user-facing locators.

Step 4: Managing asynchronicity
This is the biggest difference. Cypress manages its command queue internally. In Playwright every
step must be explicitly preceded by await.

3. Code Standard Example (Refactoring)
This is what your new Page Object class standard should look like:
TypeScript
import { Page, Locator, expect } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly submitButton: Locator;

  constructor(page: Page) {
    this.page = page;
    // We use getBy... instead of CSS selectors
    this.emailInput = page.getByLabel('Email address');
    this.submitButton = page.getByRole('button', { name: 'Log in' });
  }

  async login(email: string): Promise<void> {
    await this.emailInput.fill(email);
    await this.submitButton.click();

    // Playwright automatically waits for action stability (Auto-waiting)
  }
}

4. Eliminating Flakiness (Test resilience)
To ensure stability ("flakiness-free"), we will apply the following rules:
Web-First Assertions: We use expect(locator).toBeVisible(), which has a built-in retry mechanism.
Isolation: Every Playwright test runs in a separate BrowserContext. Avoid sharing state
between tests.
Fixtures instead of BeforeEach: Instead of creating page instances in beforeEach, use custom
fixtures, which speeds up execution and improves readability.
```

## Quick start

The tests run against a live Juice Shop on port 3000. The suite was written for the **v13-era app**
(the original Cypress suite targeted it), so pin the version — several challenge tests depend on
version-specific data and break on newer releases:

```bash
docker run --rm -p 3000:3000 bkimminich/juice-shop:v13.0.0
```

Then, in this repo:

```bash
npm install
npx playwright install     # REQUIRED: downloads the browser binaries pinned to Playwright 1.58
npm test                   # runs Chromium + Firefox; or: npm run test:chromium
npm run report             # open the HTML report
```

> ⚠️ `npx playwright install` is not optional. A fresh checkout fails 100% of tests with
> `browserType.launch: Executable doesn't exist …` until the pinned browser binaries are downloaded.

Config knobs live in [playwright.config.ts](./playwright.config.ts): `JUICE_SHOP_URL`
(default `http://localhost:3000`), 2 retries, 30 s per-test timeout, Chromium + Firefox projects.

## Test results (the honest part)

Measured on Chromium against a local **Juice Shop v13.0.0** (the version the original Cypress suite
targeted), `npx playwright test --project=chromium`.

Three measurements: the **raw AI output** (commit `4347f4b`), after **fix #1** — repairing the two
never-matching locators and missing waits in `FirstVisitPopups.dismissAllPopups()` (~15 lines in one
file), and after **fix #2** — correcting one wrong link id in `RegistrationPage` (1 line). No test
logic was touched in either fix:

| Result | Raw AI output | After fix #1 (popups) | After fix #2 (+register link) |
|--------|--------------|-----------------------|-------------------------------|
| ✅ Passed | **24 / 89** (~27%) | **57 / 89** (~64%) | **73 / 89** (~82%) |
| ⚠️ Flaky (passed on retry) | 1 | 4 | 2 |
| ❌ Failed | 64 | 28 | 14 |
| ⏱️ Suite duration | 12.6 min | 10.7 min | **6.0 min** |

Per spec file:

| Spec | Raw | Fix #1 | Fix #2 |
|------|-----|--------|--------|
| `2-stars-challenges.spec.ts` | 3 / 11 | **11 / 11** | **11 / 11** |
| `3-stars-challenges.spec.ts` | 4 / 14 | 13 / 14 | 13 / 14 |
| `homepage.spec.ts`        | 10 / 16 | 13 / 16 | 13 / 16 |
| `1-star-challenges.spec.ts` | 6 / 11 | 10 / 11 | 10 / 11 |
| `login.spec.ts`           | 0 / 11 | 9 / 11 | 9 / 11 |
| `products.spec.ts`        | 2 / 11 | 5 / 11 | 5 / 11 |
| `register.spec.ts`        | **0 / 15** | **0 / 15** | **14 / 15** |

Two fixes totalling ~16 lines across two page objects took the suite from 27% to 82% passing and
halved its runtime. That is the shape of the AI output's failure: not diffuse low quality, but a few
sharply-defined defects with a huge blast radius — invented selectors that never matched the real
DOM, hidden behind a silent `try/catch`.

### What actually works

- Most of `homepage.spec.ts` — these tests are written defensively (guarded `if (visible)` checks,
  tolerant assertions), so they survive.
- Read-only / API-style challenge checks that don't depend on clicking through the UI shell
  (score board, FTP access, exposed metrics, redirect allowlist, hidden endpoints).

### What broke

- **The popup helper that sank two-thirds of the suite** (fixed — the one manual fix in this repo).
  In the raw output every `login`/`register` test and most challenge tests died on:
  ```
  TimeoutError: locator.click: Timeout 10000ms exceeded.
    <div class="cdk-overlay-backdrop …"> … subtree intercepts pointer events
    at HeaderBar.clickAccountButton (src/pages/HeaderBar.ts:29)
  ```
  Root cause, in `FirstVisitPopups`: **both dismiss locators never matched anything on the real
  app**. The AI guessed `[mat-dialog-close]` for the welcome dialog (the actual button is
  `button[aria-label="Close Welcome Banner"]`) and `[mat-button]` with text "Accept" for the cookie
  banner (actually `a[aria-label="dismiss cookie message"]`, "Me want it!"). On top of that,
  `dismissAllPopups()` checked `isVisible()` instantly instead of waiting for the async dialog, and
  never waited for the overlay backdrop to detach. So dismissal silently did nothing, the backdrop
  swallowed every later click (442 such errors in one run), and the `try/catch` around the call hid
  it all. Fixing this one helper took the suite from 24 to 57 passing.
- **Inverted navigation link killed all of `register`** (fixed — manual fix #2, one line). All 15
  register tests navigate to the login page and then call `clickLoginLink()`, which clicked
  `#alreadyACustomerLink`. That id exists only on the *registration* page (it leads back to login);
  the login page has `#newCustomerLink`. The AI wired the navigation backwards — every register test
  timed out on a locator that cannot exist where it looked for it. Ironically the AI had already
  written the correct selector in `LoginPage.registrationLink`; changing `RegistrationPage.ts:38` to
  match took register from 0/15 to 14/15.
- **Sloppy assertion locator (strict mode violation), left unfixed.** The one remaining register
  failure is in the test itself: the security-question *selection works*, but the verification at
  `register.spec.ts:137` queries `[class*="mat-select-value"]`, which matches two elements
  (a `div` and the `span` inside it), so Playwright's strict mode rejects it. A `.first()` would fix
  it — kept as-is as an example of a third class of AI defect: assertions that fail on correct
  application behaviour.
- **Method that was never defined:** `homepage.spec.ts:116` calls `headerBar.typeInSearchInput(...)`,
  but `HeaderBar` only defines `fillInSearchInput`. The AI invented a call site for a method it
  never wrote.
- **Wrong URL assumption:** `homepage.spec.ts:16` asserts `toHaveURL(/.*home/)`, but Juice Shop
  lands on `/#/` — so "load the homepage successfully" fails.
- **Unused-symbol noise** that trips `tsc --noEmit` (e.g. an unused `captchaData` in the 1-star
  spec, an unused `TestDataGenerator` import in the products spec) and a `tsconfig.json` `references`
  entry pointing at a non-existent `tsconfig.node.json`. None of these block `npx playwright test`
  (Playwright transpiles per-file), but they show the AI left dead code and a broken TS project ref.
- Various challenge tests fail on real app timing/flow that the generated flows don't handle.

_All of the above is left in place on purpose. If any of it is fixed later, the frozen tag preserves
this original state._
